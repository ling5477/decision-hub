# 目标

为当前前端功能建立可重复执行的 Playwright 端到端回归验证，确保关键链路可用。

# 固定技术栈约束

1. 测试框架固定为 Playwright。
2. 页面为 React Router 驱动，路由进入必须纳入验证。
3. 服务端数据由 TanStack Query 驱动，测试中必须考虑异步请求、缓存刷新、mutation 后数据回刷。
4. 页面 UI 基础组件为 Ant Design，测试必须避免依赖其内部随机 class。
5. 选择器优先使用：
    - 可访问角色
    - 文本
    - label
    - 明确的 data-testid
6. 禁止大量使用 `waitForTimeout`。

# 适用范围

适用于：

- 登录流程回归
- 页面进入
- 列表加载
- 查询筛选
- 分页
- 表单提交
- 新增 / 编辑 / 删除
- 弹窗确认
- 抽屉编辑
- 详情查看
- 成功 / 失败提示验证

不适用于：

- 单元测试
- 像素级视觉对比
- 性能压测

# 必须遵守

1. 先列回归范围，再写或改测试，再执行。
2. 必须围绕真实业务链路，不写空洞用例。
3. 失败时必须说明失败步骤和失败位置。
4. 优先复用现有 fixtures、登录逻辑、test utils。
5. 对 TanStack Query 驱动页面，要等待数据加载稳定后再断言。
6. 对 Antd 组件交互，要考虑：
    - Modal 弹出
    - Drawer 展开
    - Select 下拉选择
    - Table 数据刷新
    - message / notification 提示出现
7. 必须优先使用稳定选择器。
8. 不允许依赖 Antd 内部 class 名称。

# 默认优先覆盖项

在没有更具体要求时，优先覆盖：

1. 路由可进入
2. 页面初始数据可加载
3. 查询 / 筛选可生效
4. 新增 / 编辑 / 删除流程可执行
5. Antd Modal / Drawer 正常开关
6. 提交成功后 message / table 刷新正确
7. 提交失败后错误提示正确

# 执行步骤

## 第一步：确定回归范围

先明确本次覆盖：

- 哪个页面
- 哪条核心业务链路
- 是否涉及列表、详情、表单、弹窗、抽屉
- 是否需要登录前置

## 第二步：检查现有 Playwright 结构

确认：

- tests 目录
- fixtures
- page object（如存在）
- baseURL
- 登录辅助逻辑
- 公共选择器
- 截图和 trace 配置

## 第三步：编写或更新用例

编写时遵循：

1. 操作步骤贴近真实用户。
2. 等待条件明确：
    - query 完成
    - DOM 稳定
    - message 出现
    - table 刷新
3. 断言关键结果，而不是只断言页面没崩。
4. 必要时增加 `data-testid`，但不要泛滥。

## 第四步：执行并记录

记录：

- 通过用例
- 失败用例
- 失败步骤
- 失败原因
- 阻塞点

## 第五步：形成结论

输出必须明确：

- 哪些通过
- 哪些失败
- 失败是代码问题、环境问题还是测试数据问题
- 是否达到当前回归要求

# 输出格式

每次使用本 skill，输出必须按下面结构组织：

1. 回归范围
2. 参考测试结构
3. 新增/修改的测试文件
4. 用例清单
5. 执行结果
6. 失败项说明
7. 结论
8. 后续补充项

# 质量标准

合格回归结果必须满足：

- 用例围绕关键链路
- 可重复执行
- 失败可定位
- 适配 React Router / TanStack Query / Antd 页面行为
- 结论明确

# 禁止事项

- 禁止只生成测试思路不写测试
- 禁止只验证页面渲染，不验证业务结果
- 禁止大量 `waitForTimeout`
- 禁止使用脆弱选择器依赖 Antd 内部 class
- 禁止失败后不给步骤和原因
