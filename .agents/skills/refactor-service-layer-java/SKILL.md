# 目标

在不改变现有业务结果的前提下，收口 service 层结构，提升可维护性、可测试性和可演进性。

# 适用范围

适用于：

- 巨型 Service 拆分
- 编排逻辑与领域逻辑分离
- 重复校验逻辑抽取
- DTO / DO / VO 转换收口
- 事务边界收口
- 包结构整理
- 命名收敛

不适用于：

- 纯 bugfix
- 全系统大重构
- 没有明确问题的小改动场景

# 必须遵守

1. 先确认现状问题，再做拆分。
2. 重构必须以保持行为一致为前提。
3. 必须优先补测试或确认现有测试护栏。
4. 不允许一边重构一边悄悄改业务语义。
5. 事务边界调整必须明确说明。
6. 抽公共逻辑必须基于真实重复，不做虚假抽象。
7. 包结构、类名、方法名必须体现真实职责。
8. 优先小步重构，不做一次性推翻。
9. 输出必须说明拆分前问题、拆分后结构、行为保持依据。
10. 涉及跨模块依赖时，必须说明边界变化。

# 执行步骤

## 第一步：识别现状问题

重点检查：

- 方法是否过长
- 类职责是否过多
- 事务注解是否散乱
- 校验逻辑是否重复
- 编排与领域规则是否混杂
- DTO / Entity 转换是否到处散落
- 是否存在难测代码
- 命名是否失真

## 第二步：设计拆分方案

优先拆分为：

- 应用编排服务
- 领域规则服务
- 查询服务
- 命令服务
- 校验器 / 组装器 / 转换器
- 持久化适配层

拆分必须贴合现有项目实际，不机械套模式。

## 第三步：实施小步重构

顺序建议：

1. 补测试
2. 提炼方法
3. 抽局部类
4. 收口事务边界
5. 调整命名与目录
6. 清理重复逻辑

## 第四步：验证行为一致

至少验证：

- 关键业务流程不变
- 事务行为不变
- 关键异常路径不变
- 核心输入输出不变

# 输出格式

每次使用本 skill，输出必须按下面结构组织：

1. 现状问题
2. 重构目标
3. 拆分方案
4. 修改文件
5. 事务与边界说明
6. 行为保持依据
7. 验证步骤
8. 风险说明

# 质量标准

合格重构必须满足：

- 结构更清晰
- 业务行为不变
- 命名更准确
- 测试可跟上
- 不引入新耦合

# 禁止事项

- 禁止为了“看起来高级”而过度设计
- 禁止边重构边暗改需求
- 禁止一次性改太大导致无法验证
- 禁止把简单逻辑拆成过多空壳类
