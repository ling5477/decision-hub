# 目标

从模块级别审查 Spring Boot 后端的结构健康度，发现边界不清、依赖混乱、配置散乱、装配隐患和分层被打穿的问题。

# 适用范围

适用于：

- 多模块 Spring Boot 项目
- 单模块但分层较复杂的项目
- 开发中后期架构收口
- 模块扩张后健康检查
- 合并前架构 review

不适用于：

- 单个小 bug 修复
- 单个 endpoint 开发
- 纯 DDL 审查

# 必须遵守

1. 先按现有项目目标理解模块边界，再审查问题。
2. 审查基于项目当前阶段，不搞脱离现实的大而全架构空谈。
3. 必须区分阻塞性问题和可后续优化项。
4. 必须检查模块职责、依赖方向、Bean 关系、配置边界。
5. 必须检查是否存在循环依赖、分层打穿、模块职责重叠。
6. 必须给出明确结论。
7. 不允许只有泛泛建议，必须落到具体模块、包或类。
8. 审查建议必须可执行。

# 审查维度

## 1. 模块职责
- 每个模块是否有清晰职责
- 是否存在职责重叠
- 是否存在“万能模块”

## 2. 依赖方向
- 依赖是否单向清晰
- 是否出现低层反向依赖高层
- 是否有循环依赖风险

## 3. 分层健康度
- controller 是否打穿 repository
- service 是否承担过多持久化细节
- infra 是否侵入业务层

## 4. Bean 与装配
- 自动装配是否明确
- Bean 命名与注入是否清晰
- 是否存在条件装配不稳定
- 是否存在隐式覆盖

## 5. 配置项
- 配置项是否集中
- 是否存在魔法字符串
- 是否缺少配置说明
- profile 行为是否清楚

## 6. 启动链路
- 启动类位置是否合理
- 扫描范围是否准确
- 是否有不必要的大范围扫描
- 启动失败风险点是否明确

# 输出格式

每次使用本 skill，输出必须按下面结构组织：

1. 审查范围
2. 模块职责审查
3. 依赖与分层审查
4. Bean / 装配审查
5. 配置项审查
6. 主要问题
7. 修改建议
8. 总体结论

# 审查结论标准

- 可继续推进
- 修改后继续推进
- 不建议继续叠加开发

结论必须三选一。

# 质量标准

合格审查必须满足：

- 问题具体
- 影响明确
- 建议可执行
- 结论明确

# 禁止事项

- 禁止只讲概念，不落到代码结构
- 禁止把个人偏好包装成硬性问题
- 禁止不给结论
