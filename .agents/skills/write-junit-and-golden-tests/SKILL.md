# 目标

为现有 Java 后端逻辑建立回归护栏，优先保证核心业务逻辑和关键输入输出在后续改动中不被破坏。

# 适用范围

适用于：

- Service 层业务逻辑测试
- 状态机测试
- DTO / JSON 序列化测试
- Repository / mapper 行为测试
- golden case 输入输出回归
- bugfix 后补测试
- 重构前建立保护测试

不适用于：

- 前端 E2E
- 性能压测
- 纯数据库 DDL 评审

# 必须遵守

1. 先找现有测试风格，再补测试。
2. 优先测业务行为，不只测 getter/setter。
3. 优先测边界值、异常分支、状态流转、幂等行为。
4. golden case 要体现真实输入输出，不要做无意义样板数据。
5. 测试命名要能表达业务语义。
6. 涉及 bug 修复时，必须补能复现该 bug 的测试。
7. 不允许只为提高覆盖率写无价值测试。
8. 测试必须可重复执行，不能依赖不稳定外部环境。
9. 集成测试要尽量最小化依赖范围。
10. 输出必须说明测试覆盖了什么、还没覆盖什么。

# 执行步骤

## 第一步：识别测试目标

先明确要保护的是：

- 哪个类 / 方法 / 模块
- 哪条业务规则
- 哪个异常分支
- 哪个状态流转
- 哪类输入输出格式
- 哪个曾经出过的 bug

## 第二步：选择测试层级

按目标决定测试类型：

- JUnit 单元测试：纯逻辑
- SpringBootTest / slice test：集成行为
- golden case：固定输入输出回归
- repository 测试：SQL / 持久化行为
- 序列化测试：JSON / envelope / contract

## 第三步：构建测试用例

至少覆盖：

- 正常路径
- 关键边界值
- 空值 / 缺失值
- 非法状态
- 重复调用 / 幂等
- 异常输入
- 回归案例

## 第四步：组织测试数据

遵循：

1. 测试数据尽量最小。
2. 命名体现语义。
3. golden case 使用稳定样本。
4. 不依赖脆弱顺序和当前时间，必要时固定时钟。

## 第五步：执行并总结

输出要说明：

- 新增测试文件
- 覆盖点
- 执行结果
- 仍存在的空白区域

# 输出格式

每次使用本 skill，输出必须按下面结构组织：

1. 测试目标
2. 覆盖范围
3. 新增/修改的测试文件
4. 关键用例说明
5. golden case 说明
6. 执行结果
7. 未覆盖项
8. 后续建议

# 质量标准

合格测试必须满足：

- 命名清晰
- 业务语义明确
- 边界覆盖合理
- 可重复执行
- 能保护关键行为
- 不只是形式化补测试

# 禁止事项

- 禁止只断言不报错
- 禁止堆砌无意义 mock
- 禁止只测 happy path
- 禁止为了过测试而改业务语义
