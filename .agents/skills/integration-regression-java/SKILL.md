# 目标

把“能编译”提升为“关键链路能闭环跑通”，为开发中后期的后端模块建立集成回归护栏。

# 适用范围

适用于：

- Controller -> Service -> Repository 闭环验证
- 数据库读写闭环验证
- 状态流转验证
- 幂等验证
- 事务回滚验证
- 本地 Docker 依赖联调
- 关键功能上线前回归

不适用于：

- 纯单元测试
- 压测
- 前端 E2E

# 必须遵守

1. 先定义回归范围，再执行验证。
2. 回归优先覆盖关键业务闭环，而不是追求大而全。
3. 测试或验证步骤必须可复现。
4. 必须区分代码问题、环境问题、数据准备问题。
5. 对涉及数据库写入的验证，必须说明前置数据和清理策略。
6. 事务、幂等、状态流转类验证必须有明确断言。
7. 无法执行时，必须说明阻塞点，不允许含糊带过。

# 执行步骤

## 第一步：确定回归范围

优先确认是否覆盖：

- 接口可达
- 请求参数可通过
- Service 链路正常
- 数据可写入 / 可读取
- 状态正确流转
- 异常情况下能回滚
- 重复调用是否幂等
- 关键审计 / 事件 / 日志是否落地

## 第二步：准备依赖与数据

明确：

- 本地数据库
- Docker 依赖
- 配置 profile
- 测试账户 / 测试数据
- 前置初始化
- 清理策略

## 第三步：执行验证

验证方式可包括：

- JUnit 集成测试
- SpringBootTest
- Testcontainers
- 本地接口调用
- 脚本调用
- golden 输入输出检查

## 第四步：汇总结论

说明：

- 哪些链路通过
- 哪些失败
- 失败原因是什么
- 是否达到当前阶段验收要求

# 输出格式

每次使用本 skill，输出必须按下面结构组织：

1. 回归范围
2. 前置条件
3. 验证方式
4. 执行结果
5. 失败项说明
6. 风险点
7. 结论

# 结论标准

- 通过
- 修复后再回归
- 当前不通过

结论必须三选一。

# 质量标准

合格回归必须满足：

- 范围明确
- 前置明确
- 结果明确
- 失败可定位
- 结论明确

# 禁止事项

- 禁止只说“本地测过”
- 禁止不写前置条件
- 禁止失败后不区分环境问题和代码问题
