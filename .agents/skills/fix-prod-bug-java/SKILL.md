# 目标

对 Java 后端问题执行“可复现、可定位、可修复、可验证”的最小正确修复。

本 skill 的重点不是顺手重构，而是先完成问题闭环：
1. 复现
2. 定位根因
3. 最小修复
4. 验证结果
5. 说明风险

# 适用范围

适用于：

- Controller / Service / Repository 报错
- 运行时异常、空指针、类型转换异常
- 参数绑定错误
- JSON 序列化/反序列化问题
- JDBC / MyBatis / JPA 查询异常
- 事务回滚异常
- 并发冲突
- 幂等失效
- 状态机流转错误
- 配置错误导致的启动或调用异常

不适用于：

- 全新功能开发
- 大规模重构
- 表结构整体改造
- 无证据的大范围架构推翻

# 必须遵守

1. 先复现，再修复，禁止凭感觉改代码。
2. 必须说明根因，不能只改现象。
3. 优先最小正确修改，禁止借修 bug 顺手做大重构。
4. 必须明确影响范围。
5. 必须说明验证方式。
6. 涉及事务、并发、幂等、缓存时，必须明确是否存在连带风险。
7. 修复时优先对齐现有项目风格和异常处理约定。
8. 禁止用吞异常、硬编码、扩大 catch 范围来掩盖问题。
9. 禁止为了避免报错直接删除校验、事务、状态判断。
10. 无法完整复现时，也必须给出证据链和最可能根因。

# 执行步骤

## 第一步：收集问题信息

优先确认：

- 报错接口 / 方法 / 类
- 触发条件
- 输入参数
- 实际报错日志
- 完整异常栈
- 是否必现
- 是否与并发、数据状态、数据库记录、环境配置有关
- 是否最近刚做过相关改动

## 第二步：建立证据链

重点查看：

- Controller 入参
- DTO / VO / Entity 映射
- Service 编排链路
- Repository / Mapper / SQL
- 事务注解与传播行为
- 日志上下文
- 状态流转分支
- 幂等键或唯一约束
- 时间、空值、边界条件
- 最近相关提交

## 第三步：确认根因

根因必须明确归类，例如：

- 参数为空且未保护
- DTO 字段与 JSON 不匹配
- SQL 条件错误
- 数据状态不满足前置条件
- 事务边界错误
- 并发下重复提交
- 幂等键错误
- 枚举状态判断漏分支
- 配置项未生效
- 时区 / 时间类型转换错误

## 第四步：实施最小修复

遵循：

1. 优先修根因点。
2. 不改无关模块。
3. 仅在确实必要时抽公共逻辑。
4. 修复后保持命名、日志、异常风格一致。
5. 需要新增保护时，保护条件要准确，不可泛化。

## 第五步：验证闭环

至少验证：

- 原问题是否消失
- 主要正常路径是否仍可工作
- 相邻分支是否受影响
- 事务与回滚是否符合预期
- 并发 / 幂等类问题是否有护栏

# 输出格式

每次使用本 skill，输出必须按下面结构组织：

1. 问题现象
2. 复现条件
3. 根因分析
4. 修改文件
5. 修复说明
6. 影响范围
7. 验证步骤
8. 风险说明

# 质量标准

合格修复必须满足：

- 根因明确
- 改动聚焦
- 验证清晰
- 不引入明显新问题
- 不靠掩盖问题通过编译

# 禁止事项

- 禁止只给猜测，不给证据链
- 禁止只改表层，不改根因
- 禁止为了修一个 bug 改整个模块结构
- 禁止吞异常
- 禁止删除校验逻辑来绕过问题
