# 目标

对数据库结构变更进行可落地、可回放、可维护的审查，确保迁移脚本安全、语义清晰、执行顺序合理、兼容现有系统。

# 适用范围

适用于：

- 新建表
- 改表结构
- 增删字段
- 索引调整
- 唯一约束 / 外键 / 检查约束
- 默认值调整
- comment 回填
- 历史数据 backfill
- Flyway / Liquibase migration 审查

不适用于：

- 单纯业务代码 review
- ORM 纯映射 review
- 无数据库变更的功能开发

# 必须遵守

1. 先理解业务语义，再看表和字段。
2. migration 必须可执行、可回放、顺序明确。
3. 字段命名、表命名、索引命名必须与项目规范一致。
4. 必须检查是否存在冗余字段、歧义字段、过宽字段。
5. 必须检查非空、默认值、唯一约束是否合理。
6. 必须检查索引是否支持真实查询路径，而不是凭感觉加索引。
7. backfill 脚本必须考虑幂等、分批、锁影响、失败恢复。
8. 注释应覆盖关键表和关键字段，不允许核心字段无语义说明。
9. 不允许把应用层约束错误地下沉为数据库约束，也不允许该由数据库兜底的约束缺失。
10. 审查结论必须明确“可通过 / 修改后通过 / 不建议合并”。

# 执行步骤

## 第一步：确认本次变更目标

先明确：

- 是新建表还是改表
- 支撑哪个业务能力
- 新增查询路径是什么
- 写入路径是什么
- 是否涉及历史数据兼容
- 是否需要回填
- 是否需要读写双兼容

## 第二步：审查 DDL 设计

逐项检查：

- 表名是否准确
- 主键策略是否合理
- 字段类型是否合适
- varchar / numeric / timestamp 精度是否合适
- 是否滥用 text/jsonb
- 是否存在状态字段语义不清
- 是否需要审计字段（created_at / updated_at / version 等）
- 是否需要租户、账号、trace 维度字段
- comment 是否完整

## 第三步：审查约束与索引

重点检查：

- 主键
- 唯一键
- 外键是否真的需要
- 检查约束是否合理
- 组合索引顺序是否匹配查询条件
- 是否存在重复索引
- 是否存在写放大严重的无效索引
- 是否缺少关键查询索引

## 第四步：审查 migration 与 backfill

重点检查：

- 文件命名与顺序
- 是否包含危险 DDL
- 是否会锁大表
- 是否可分步上线
- 是否兼容旧代码
- backfill 是否幂等
- backfill 是否支持重跑
- 是否需要灰度或双写双读过渡

## 第五步：给出结论

结论必须分级：

- 可通过
- 修改后通过
- 不建议合并

# 输出格式

每次使用本 skill，输出必须按下面结构组织：

1. 变更目标
2. 表/字段设计审查
3. 约束与索引审查
4. migration / backfill 审查
5. 主要问题
6. 修改建议
7. 风险点
8. 审查结论

# 质量标准

合格审查必须满足：

- 业务语义清楚
- 字段设计清楚
- migration 安全
- 索引贴合查询路径
- 风险明确
- 结论明确

# 禁止事项

- 禁止只看语法是否正确，不看业务语义
- 禁止只说“建议加索引”却不说明查询路径
- 禁止忽略回填风险
- 禁止忽略兼容期读写行为
